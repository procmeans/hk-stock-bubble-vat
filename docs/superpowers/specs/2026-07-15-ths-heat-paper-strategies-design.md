# 同花顺热度双策略模拟盘设计

日期：2026-07-15  
状态：已完成口头确认，待书面审阅

## 目标

在现有三个模拟账户旁新增两个 A 股纸面模拟账户：

- `a_ths_heat`：按同花顺当日个股热度从高到低选股；
- `a_ths_heat_rise`：按同花顺个股热度排名环比增长率从高到低选股。

两个账户都使用 Top 20 等权、每 2 个交易日调仓、T 日收盘信号于
T+1 日收盘成交、单边 20bp 成本和 ¥100,000 初始资金。目标是进行前向、
可审计的单因子对照实验，不预设热度因子一定有效。

## 已验证的数据接口

使用同花顺 iFinD HTTP API 的智能选股接口：

```text
POST https://quantapi.51ifind.com/api/v1/smart_stock_picking
{"searchstring": "<查询语句>", "searchtype": "stock"}
```

官方文档：
https://quantapi.51ifind.com/gwstatic/static/ds_web/quantapi-web/help-center/manual.html

已用仓库现有 `THS_HTTP_REFRESH_TOKEN` 做只读验证：

- `个股热度排名前20` 返回股票代码、股票简称、个股热度、排名、名次和排名基数；
- 指定交易日的 `个股热度排名环比增长率排名前20` 返回该日增长率和排名；
- 返回列名带动态日期后缀，例如 `个股热度[20260715]`，解析器不能依赖完整固定列名；
- 显式指定行情面板最新交易日可以避免自然日、时区和派生字段更新节奏造成的日期错位。

每个调仓日使用以下两条语义查询，其中日期取 A 股行情面板的最新交易日：

```text
YYYY年M月D日个股热度排名前20
YYYY年M月D日个股热度排名环比增长率排名前20
```

## 架构与职责

### `alpha101/ths_http.py`

新增 `smart_stock_picking(searchstring, searchtype="stock", ...)` 薄封装，复用现有
access token 获取、HTTP POST、API 错误处理和表格归一化逻辑。该层不包含热度策略语义。

### `strategies/ths_heat.py`

该模块集中负责热度因子：

1. 根据 `signal_date` 和 `kind`（`heat` / `heat_rise`）生成指定日期查询语句；
2. 调用 `ths_http.smart_stock_picking`；
3. 用列名前缀匹配动态日期列，并解析列名中的日期；
4. 校验返回数据日期与请求交易日相同；
5. 标准化为 `date, strategy, rank, ticker, name, factor_value`；
6. 剔除代码或因子值无效的记录，保留前 `top_n` 条；
7. 对最终具有有效价格的股票生成等权目标。

解析器必须对列顺序无假设。缺少代码列、目标因子列、日期不匹配或空结果均产生明确异常，
由模拟盘编排层按单账户隔离处理。

### `strategies/paper.py`

新增参数：

```python
THS_HEAT_PARAMS = {"top_n": 20, "rebalance": 2}
```

并把 `ths_heat`、`ths_heat_rise` 注册到 `DEFAULT_PARAMS` 和目标权重分发逻辑。

`run-market --market a` 的数据流调整为：

1. 按现有方式获取 A 股基础行情面板，并保留流动性前 500 作为固定基准池；
2. 用行情面板最新日期判断每个热度账户是否将在本次步进触发调仓；
3. 仅在至少一个热度账户到期时请求对应因子，非调仓日不消耗热度接口配额；
4. 对热榜股票中不在基础面板的代码补抓历史行情并合并到可交易面板；
5. 将每个账户自己的预取目标传给纯步进逻辑，避免在 `step()` 内进行网络请求；
6. 所有 A 股账户仍共用基础行情抓取；两个热度查询各最多调用一次，不随账户数重复；
7. 各账户独立步进、持久化和报告状态。

为避免热榜股票扩展交易池后改变比较基准，面板提供独立的基准收盘价集合。
`step()` 使用固定的流动性前 500 计算 `bench_nav`，而估值与成交使用包含持仓、挂单和热榜股票的
可交易价格集合。该变化保持现有账户策略逻辑不变，并使账户之间的基准口径一致。

## 交易规则

两个账户采用完全相同的执行参数，使结果差异只来自因子：

- 初始资金：¥100,000；
- 持仓：因子 Top 20，有效标的等权；
- 调仓：每 2 个交易日；
- 时序：T 日收盘形成目标，T+1 日收盘成交；
- 费用：成交绝对金额的单边 20bp；
- 基准：流动性前 500 股票等权逐日收益；
- 纸面简化：允许碎股，不模拟 100 股整手、涨跌停无法成交和市场冲击。

若少于 20 只股票具有有效因子和有效收盘价，则在剩余股票之间重新等权，权重总和为 1。
无有效股票时不生成新挂单，保留当前持仓，并在下一个交易日重试调仓信号。

`rebalance=2` 的准确含义是：T 日生成信号、T+1 收盘建仓，持有两个收盘到 T+3 收盘换入下一组；
不是每两个自然日运行。

## 信号审计与幂等

新增 `paper/ths_heat_signals.csv`，成功记录包含：

```text
date,strategy,rank,ticker,name,factor_value,status,error
```

- 成功查询为每个策略保存最终使用的排名记录，`status=ok`；
- 查询、解析或补价失败保存一条 `status=error` 记录和经过清理的错误摘要；
- 唯一键为 `date + strategy + rank + ticker + status`；
- 同日工作流重跑不得重复追加信号、净值或订单；
- 凭证、access token、refresh token 和原始响应头绝不进入日志或仓库。

信号流水用于回答“该账户当日为何持有这些股票”，不作为回测数据仓库。

## 故障隔离

- 一个热度查询失败时，仅对应账户本次不生成新目标；另一个热度账户继续执行；
- 两个热度查询都失败时，现有 A 股 Alpha101 和流动性等权账户仍正常记录净值与调仓；
- 热度账户在信号失败日继续按已有持仓计价，不清仓，调仓计数保持到期状态，下一交易日重试；
- 某些热榜股票无法补得有效收盘价时，仅剔除这些标的并重新等权；
- 基础 A 股行情面板整体失败时沿用现有行为：命令非零退出，不写入任何虚假价格或信号；
- API 返回成功但日期不是请求交易日时视为错误，禁止使用陈旧因子。

## 测试与验证

### 单元测试

`alpha101/ths_http.py`：

- `smart_stock_picking` 发送正确 endpoint 和 payload；
- 复用 access token 和错误处理。

`strategies/ths_heat.py`：

- 两类查询语句包含明确交易日；
- 动态日期列在列顺序变化时仍正确解析；
- 返回日期不匹配、缺列、空结果和非法数值时行为明确；
- 代码标准化、Top N 截断和有效标的重新等权正确。

`strategies/paper.py`：

- 两种策略分发到各自预取目标；
- `rebalance=2` 的信号、成交和再平衡日期正确；
- T+1 成交、20bp 成本和会计恒等保持；
- 热榜代码在基础池外时会补价且能成交；
- 交易池扩展不改变流动性 500 基准收益；
- 单个接口失败不阻断其他账户；
- 失败后保持到期状态并在下一交易日重试；
- 信号、净值和订单流水同日重跑不重复；
- 非调仓日不调用热度接口。
